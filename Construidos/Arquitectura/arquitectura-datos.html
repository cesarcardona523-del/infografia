Catálogo de Referencia — Arquitectura de Datos

9 Patrones que Todo Arquitecto Debe Conocer

Un panorama comparativo — cada patrón resuelve un problema distinto de latencia, gobernanza y costo.

HOJA 01/01
ESCALA — CONCEPTUAL

01Data Warehouse

Sources→ETL→Warehouse

Repositorio centralizado para datos estructurados, optimizado para análisis histórico.

LatenciaBatch
GobernanzaCentralizada

02Data Lake

Sources→Ingestion→Lake→Analysis

Almacén flexible para datos crudos y estructurados a gran escala, bajo costo.

LatenciaBatch
GobernanzaCentralizada

03Lambda Architecture

Batch+Speed→Serving→Queries

Combina procesamiento por lotes y en tiempo real para vistas completas.

LatenciaHíbrida
GobernanzaCentralizada

04Kappa Architecture

Speed Layer→Serving→Queries

Procesa todo como flujo continuo, simplificando la arquitectura Lambda.

LatenciaStreaming
GobernanzaCentralizada

05Data Mesh

Domain Products→Self-Service & Governance

Enfoque descentralizado por dominios, tratando los datos como producto.

LatenciaVariable
GobernanzaPor dominio

06Data Lakehouse

Sources→Lakehouse→BI & AI

Unifica Data Lake y Warehouse, con gestión transaccional (ACID).

LatenciaHíbrida
GobernanzaCentralizada

07Data Fabric

Sources→Fabric→Consumers

Capa de gestión inteligente que conecta datos dispersos automáticamente.

LatenciaVariable
GobernanzaFederada

08Event-Driven

Producers→Event Broker→Consumers

Arquitectura reactiva: los servicios se comunican mediante eventos asíncronos.

LatenciaStreaming
GobernanzaDescentralizada

09Streaming

Streaming Sources→Processing→Real-time

Procesamiento continuo de datos en movimiento para insights inmediatos.

LatenciaReal-time
GobernanzaCentralizada
Cómo elegir — estos patrones no son excluyentes: un Lakehouse moderno suele incorporar ingesta event-driven, y Kappa es Lambda simplificada cuando el batch layer deja de ser necesario. La elección real depende de la latencia requerida, la gobernanza deseada (centralizada vs. por dominio) y el presupuesto de cómputo/almacenamiento.
